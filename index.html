    Crack 1 — The Missing “When” Layer

    Jonala caught it: in v1 the LLM is a pure extraction proxy. Every message gets shoved through the weather workflow — even “good morning.” Nothing decides whether a tool call is needed at all.

    ❓ Ask the room

    You run a company front desk with one receptionist and 40 departments. The receptionist can't possibly know every department's work. How do you make sure a walk-in visitor is either answered directly or routed to the right department?

    Let the room build the answer. Target landing zone: a directory — each department's name, a one-line description of what it handles, what info it needs — and a structured routing slip.

    🔥 Feel it — the v1 pipeline, live

    v1 has no “whether.” Watch it try to extract a city from arithmetic.

    This is yesterday's flow, exactly as built: every message goes to the LLM with one job — extract a city — then straight to OpenWeatherMap. Run the presets. Feel the absurdity.

    User message
    LLM probabilistic
    OpenWeatherMap deterministic
    Reply to user
    🔧 Exercise A — be the receptionist

    For each walk-in: answer directly, or fill a routing slip?

    The routing decision is the missing layer. Make it by hand first — then we'll make the model make it.

    🔧 Exercise B — write the directory entry

    Define get_weather so a receptionist who has never seen it knows when to route to it

    Three things per entry, exactly like the department directory: a dispatchable name, a description that answers “when is this the right choice?”, and an input schema — the contract.

    city: string — declared
    required: ["city"] — enforced, not optional
    
          
    Waiting… would your receptionist know when to route to this?

      💡 The Reveal
      
      
    
        
    The directory is a tool menu. The routing slip is a structured intent.
    
        
    What the LLM needs, delivered through the system prompt, is a menu of tools — and one new instruction that upgrades it from extractor to decision maker:
    
        
    
          The new instruction
          “Either answer the user directly, or, if a tool is needed, output a structured call in exactly this JSON shape: { name, arguments }. Nothing else.”
        
    
        
    # The tool menu (lives in the system prompt / API tools param)
{
  "name": "get_weather",
  "description": "Get current weather for a city.
                  Use when the user asks about weather,
                  temperature, or conditions anywhere.",
  "parameters": {
    "type": "object",
    "properties": { "city": { "type": "string" } },
    "required": ["city"]
  }
}

# What the LLM now emits — an intent, NOT an action
{ "name": "get_weather", "arguments": { "city": "Bengaluru" } }
    
        
    The LLM never executes anything. It emits an intent; the execution environment reads the intent, runs the code, and owns everything that actually happens. Yesterday's division of labor survives intact: probabilistic decides & speaks · deterministic acts & verifies.
    
        
    
          Now — and only now — the name
          This exact pattern — a tool menu plus a structured call emitted by the model — is what every provider ships natively as function calling (or tool use). OpenAI, Anthropic, Groq: the tools parameter in their APIs is this JSON menu, standardized. You didn't learn a feature. You re-derived why the feature has the shape it has.

    # Pass 1: model decides
resp = client.chat.completions.create(
    model=MODEL,
    messages=[{"role": "user", "content": user_msg}],
    tools=[GET_WEATHER_TOOL],   # the menu from Segment 02
)
call = resp.choices[0].message.tool_calls[0]   # intent, not action

# Execution: deterministic side acts — the LLM is not involved
args = json.loads(call.function.arguments)
weather = fetch_openweathermap(args["city"])   # our L13 automation

# Pass 2: model speaks, grounded
messages += [resp.choices[0].message,
             {"role": "tool", "tool_call_id": call.id,
              "content": json.dumps(weather)}]
final = client.chat.completions.create(model=MODEL, messages=messages)

    Cracks 2 & 3 — Hallucinated Arguments, Garbage In
    
    
    Yesterday's estimate holds: roughly 90% right — and the 10% is fatal if unhandled, because the deterministic side amplifies garbage-in into garbage-out.
    

    
    
      ❓ Ask the room
      
    You cannot retrain the model — the weights are frozen. You cannot control how users type. Where in the three layers is the one place you fully control — and what do you already know how to do there?
    
      
    Target landing zone: the execution environment — and what we do to deterministic software is verify it. The Verifier's Rule applies at the boundary.

      💡 The Reveal
      
      
    
        
    The arrow that used to flow one way now cycles
    
        
    
          model proposes→
          environment executes + verifies→
          result · or error→
          model adjusts
          ↺ repeat until done
        
    
        
    Validation failures become tool messages: city 'my cousin' not found; ask the user or re-extract. The model retries with better arguments. The wrong extraction gets caught and corrected in one cycle, on screen.
    
        
    
          Plant the seed — do not name it
          The cycle the class just drew — propose → execute + verify → feed back → adjust → repeat — is the primitive the entire agents module is built from. Say only: “hold onto this loop; in a few weeks you will see it is the whole story.”

    Crack 4 — The Scale Wall
    
    
    Zoom out from one weather tool. Your workflow diagnoses need Gmail, LinkedIn, Instagram, Supabase, internal wikis. And the world has more than one model: GPT-OSS today, Claude tomorrow, a local Llama for the client who can't send data out.
    

    
    
      ❓ Ask the room
      
    If every tool needs custom glue for every model or app, how many integrations does the grid cost? And where have you seen this exact problem shape before — outside of AI?
    
      
    Target landing zone: M × N, rebuilt by every team on Earth. Prior art the class already knows: USB. HTTP. The answer to an M × N explosion is always a standard in the middle.

    🔌 USB
    Before: every peripheral shipped its own port and driver per machine. After: one plug shape, any device into any computer. M + N.
    
        
    🌐 HTTP
    Any browser talks to any server — because both speak one grammar. Nobody writes Chrome-specific web servers. M + N.
